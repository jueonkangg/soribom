# 강구현 개발 로그 (STT / 자막 파이프라인)

내가 맡은 부분: **소리를 받아서 → 말하는 구간만 잘라서 → 자막으로 바꾸기**.
어려운 말로는 STT(음성인식) pipeline이다.

---

## 2026년 7월 20일

### 오늘 한 것

**1. 내 작업 공간 만들기 (git branch)**
- `main`은 팀 공용이라 여기서 막 고치면 안 된다. 그래서 내 branch인 `guhyeon`에서 작업했다.
- 커밋에 내 이름이 찍히게 git 설정(`user.name`, `user.email`)도 했다.
- 배운 것: branch는 **각자 따로 노는 놀이터** 같은 거다. 내가 여기서 코드를 망가뜨려도 친구들 것은 안 망가진다. 다 되면 `main`으로 합친다(merge).

**2. 마이크에서 소리 받기 (`capture.py`)**
- ReSpeaker라는 4채널 mic에서 소리를 읽어 `queue`(줄서기 통)에 넣는 코드를 짰다.
- `sounddevice`가 소리를 조금씩 모아서 `callback`이라는 함수를 자동으로 불러준다. 나는 그때마다 소리를 queue에 넣기만 하면 된다.
- 이 mic는 채널이 6개인데, **채널 0번이 칩이 미리 깨끗하게 다듬어 준 소리**라 그것만 쓰기로 했다.
- 확인 방법: 3초 녹음해서 소리 크기(RMS)를 찍어보고 `.wav`로 저장해서 직접 들어봤다.
- 배운 것:
  - `queue` = 소리 데이터를 한 칸씩 넣고 빼는 통. 넣는 쪽(mic)과 쓰는 쪽(VAD)을 이어준다.
  - `callback` = 내가 부르는 게 아니라 **필요할 때 자동으로 불려지는** 함수.
  - 소리는 `float32` 숫자(-1 ~ 1)로 들어온다. RMS가 크면 소리가 큰 거다.

**3. 말하는 구간만 자르기 (`vad.py`)**
- VAD = Voice Activity Detection. **지금 사람이 말하는 중인지 아닌지**를 판단하는 것.
- `silero`라는 똑똑한 VAD 모델을 써서, 소리를 512개 샘플씩(32ms) 잘라 "말일 확률"을 매겼다.
- 확률이 높으면 모으고, 조용해지면 "한 문장 끝!" 하고 그 구간을 내보낸다.
- 처음 만들 때 앞부분이 잘리지 않게 살짝 여유(`speech_pad_ms`)를 앞에 붙였다.
- 결과: 8초 동안 문장 몇 개를 말했더니 **4개 발화로 정확히 잘라냈다.**

### 오늘 크게 막혔던 것 (그리고 어떻게 풀었나)

- 처음엔 인터넷에서 받은 silero 모델을 직접 붙였는데, **말을 해도 확률이 계속 0**이 나왔다. 소리는 분명히 크게 들어오는데(RMS 0.11) 모델만 못 알아들었다.
- 원인을 찾으려고, **이미 잘 작동하는 것(faster-whisper 안에 든 VAD)과 같은 소리로 비교**해봤다. 그건 말을 잘 찾아냈다. 즉 소리는 문제없고 내 모델 부르는 방식이 틀린 거였다.
- 그래서 **faster-whisper에 이미 들어있는 VAD로 갈아탔더니 바로 해결**됐다. 덤으로 무거운 `torch`도 안 깔아도 됐다.
- 배운 것:
  - 안 될 때는 혼자 헤매지 말고 **"되는 것"과 비교**하면 원인이 좁혀진다.
  - 이미 잘 만들어진 걸 쓰면(재사용) 시간도 줄고 버그도 준다.
  - `Jetson`(우리 컴퓨터)에서는 `torch` 깔기가 까다로워서, 되도록 `onnx`로 도는 걸 쓰는 게 좋다.

### 오늘의 결과
- `capture.py`, `vad.py` 완성하고 **실제 mic로 검증**까지 끝냄.
- 다음 할 일: **2-패스 STT (`transcriber.py`)** — 작은 model로 빠른 임시 자막, 큰 model로 정확한 교정 자막.

## 2026년 7월 21일

### 오늘 한 것: 자막 만들기(STT)와, 그걸 빠르게 만들기

**1. 소리를 글자로 (`transcriber.py`)**
- `faster-whisper`라는 음성인식(STT) 라이브러리로, 발화 덩어리를 받아 한국어 자막으로 바꿨다.
- 이미 받아둔 `small`, `medium` 두 model이 있었다. 작은 게 빠르고, 큰 게 정확하다.

**2. 너무 느린 문제 → GPU를 쓰자**
- 처음엔 `CPU`로 돌렸더니 medium이 8초 오디오에 **12초**나 걸렸다. 실시간보다 느리다!
- 원인: 우리 컴퓨터(Jetson)엔 `GPU`가 있는데, 음성인식 엔진(`CTranslate2`)이 **GPU를 못 쓰는 버전**이었다.
- 그래서 `CTranslate2`를 **직접 소스에서 CUDA(GPU용)로 컴파일**했다. 큰 작업이었지만 성공.
- 결과: medium이 12초 → **1.7초**로 빨라졌다. (GPU는 정말 빠르다)

**3. 큰 벽: medium이 메모리에 안 들어감**
- 우리 Jetson은 메모리 8GB를 `CPU`와 `GPU`가 **나눠 쓴다**. medium은 너무 커서 화면·다른 기능이랑 같이 켜면 **메모리 부족(OOM)** 으로 죽었다.
- 더 비싼 컴퓨터면 됐겠지만, 예산(총 150만원)이 정해져 있으니 어쩔 수 없다.

**4. 그럼 어떤 걸 쓸지 — 숫자로 결정했다 (측정!)**
- 대본을 정해 읽고 녹음한 뒤, 정답과 비교해 **CER(글자 오류율)** 을 쟀다. (조용할 때 / 시끄러울 때 둘 다)
- 놀라운 결과:
  - `small`도 오류율이 **약 1%**로 아주 정확했다. 심지어 **시끄러워도 그대로 정확**했다.
    (ReSpeaker mic가 잡음을 하드웨어로 걸러주기 때문 — 우리 장비의 큰 장점)
  - `medium`이 `small`보다 딱히 더 정확하지도 않았다.
- 그래서 **2-패스를 접고 `small` 하나만** 쓰기로 했다.

**5. 남은 오류를 안전하게 줄이기 (프롬프트)**
- "광합성"을 "광업성"으로 잘못 듣는 오류가 몇 개 있었다.
- `initial_prompt`에 **수업 용어들(광합성, 이산화탄소 …)을 힌트로** 줬더니, 인식이 맞는 쪽으로 정렬돼 오류율이 **1.4% → 0.0%** 가 됐다.
- 이건 없는 말을 지어내는 게 아니라 **후보를 골라주는** 것이라 안전하다. (생성 model로 문장을 고치는 방법도 있지만, 맞는 말까지 바꿔버릴 위험이 있어 안 쓰기로 했다)

### 오늘 배운 것
- **느리면 원인을 찾자.** CPU가 느리면 GPU가 있는지, 왜 못 쓰는지 파고들면 길이 나온다.
- **소스 빌드**: 필요한 기능(CUDA)이 없는 라이브러리는 직접 컴파일해서 만들 수 있다.
- **메모리는 공짜가 아니다.** 특히 Jetson은 CPU·GPU가 메모리를 나눠 쓴다.
- **결정은 느낌이 아니라 숫자로.** CER을 재보니 "큰 model이 낫겠지"라는 예상이 틀렸다.
- **정확도는 꼭 큰 model이 아니어도 된다.** 좋은 mic(하드웨어) + 똑똑한 힌트(prompt)가 model 크기를 이겼다.

### 오늘의 결과
- `transcriber.py` 완성 (`small` 단일 패스 + 도메인 prompt), **CER ~0% 실측**.
- 측정 도구(`record_test.py`, `stt_eval.py`, `test_reference.txt`)도 만들어 둠 → §8 지표(WER)에 재사용.
- **자막 파이프라인 3단계(capture → VAD → STT) 전부 완성.** 다음은 세 기능 한 화면 통합(UI는 도유 담당).
