## 1. 자막 정확도 (WER / CER)

측정: `test_reference.txt`(교실 상황 6문장, 42어절)를 녹음한 `test_clean.wav`(조용) /
`test_noisy.wav`(잡음)를, 실제 파이프라인 그대로 **VAD로 발화 분할 + small + 도메인 prompt**로 전사.

| 조건 | CER(문자오류율) | WER(어절오류율) | 비고 |
|---|---|---|---|
| **조용할 때** (clean) | **0.0%** | 16.7% | 완벽 인식 |
| **시끄러울 때** (noisy) | **0.7%** | 18.8% | "칠판에"→"칠판이" 1글자만 |

**핵심 해석 (정직하게):**
- **CER(문자 단위)가 진짜 정확도**다. 조용할 때 0.0%, 잡음에도 0.7% — 사실상 완벽.
- **WER 16.7%는 오인식이 아니라 대부분 '띄어쓰기 차이'**다. 예:
  - 정답 `빛에너지` ↔ 인식 `빛 에너지`
  - 정답 `풀어 볼까요` ↔ 인식 `풀어볼까요`
  - 정답 `읽어 오세요` ↔ 인식 `읽어오세요`
  - 정답 `이야기해 주세요` ↔ 인식 `이야기해주세요`
  한국어 띄어쓰기는 사람마다도 갈리는 부분이라, 의미 전달에는 영향이 없다.
  자막을 읽는 학생 입장에서 문제되는 실제 오류는 잡음 조건의 `칠판에→칠판이` 한 곳뿐이다.
- **잡음에도 정확도가 거의 그대로**인 게 강점이다. ReSpeaker의 XMOS 칩이 빔포밍·잡음억제를
  하드웨어에서 처리한 ch0만 쓰기 때문. (직접 잡음처리 코드 없음)
- 도메인 prompt 효과: prompt를 빼면 `광합성→광업성`, `칠판→7판`으로 틀린다(CER 1.4%).
  수업 용어를 힌트로 주면 CER 0.0%로 교정된다. 생성이 아니라 '인식 편향'이라 없는 말을 만들지 않는다.

## 2. 자막 지연 (말이 끝나고 자막이 뜰 때까지)

측정: 발화 한 덩어리를 GPU small로 전사하는 데 걸린 시간(발화당). VAD가 발화 끝을 감지한 뒤의 처리 시간.

| 조건 | 최소 | 중앙값 | 최대 |
|---|---|---|---|
| 조용할 때 | 0.48s | **0.62s** | 0.80s |
| 시끄러울 때 | 0.52s | **0.64s** | 0.74s |

- 발화가 끝나면 **약 0.6초** 뒤 자막이 뜬다. 실시간 대화를 따라가기에 충분히 빠르다.
- (여기에 VAD가 발화 끝을 확정하는 `min_silence_ms=400ms`가 더해져, 체감 지연은 ~1초 안팎.)

## 3. TTS 지연 (타이핑 완료 → 스피커에서 소리)

측정: `speaker.py`로 문장을 GPU(Supertonic 32스텝)로 합성하는 데 걸린 시간. 문장 길이별.

| 문장 | 합성시간 | 오디오 길이 | 실시간 배속 |
|---|---|---|---|
| "네." (2자) | 1.65s | 1.4s | 1.22x |
| "질문 있습니다." (8자) | 1.51s | 1.6s | 0.93x |
| "선생님, 다시 한 번 설명해 주세요." (20자) | 1.79s | 3.2s | 0.56x |
| "저는 이 문제의 답을 3번이라고 생각합니다." (24자) | 1.79s | 3.8s | 0.47x |
| "오늘 배운 광합성 내용 중에서…" (41자) | 1.98s | 6.0s | 0.33x |

- **합성 지연 중앙값 ~1.8초, 범위 1.5~2.0초.** 문장이 길어도 지연은 거의 안 늘고(고정 오버헤드 지배),
  긴 문장일수록 실시간 대비 훨씬 빠르다(0.33x).
- Enter를 누르면 **약 2초 안에** 스피커가 말하기 시작한다. 대화에 끼어들 만한 속도.
- GPU 초기화 실패(메모리 부족) 시 `speaker.py`가 CPU로 자동 폴백 → 느려지지만(32스텝 ~9초) 소리는 난다.

## 4. STT + TTS 동시 상주 (통합 OOM 검증)

측정: `integration_smoketest.py test_clean.wav` — 두 모델을 GPU에 함께 올리고 추론/합성.

- STT 로드 3.0s + TTS 로드 2.0s, 둘 다 `provider=cuda` 상주.
- STT 추론 2.29s(40초 오디오), TTS 합성 1.8s.
- **테스트 중 최저 여유 메모리 3,666 MB, Swap 사용 0 MB → OOM 없음.**
- ⚠️ 단, Chrome 등이 떠 있으면 여유가 줄어 GPU 초기화가 실패할 수 있다(과거 `CUBLAS_ALLOC_FAILED`).
  **데모 전 브라우저 등 메모리 큰 프로그램을 끌 것.** (Jetson은 CPU/GPU가 8GB RAM 공유)

## 5. 라이브 통합 실행 확인 (부분 확인)

`src/main.py` 실행 → 전체화면 UI에 **자막·방향 화살표·입력창 세 요소가 함께 표시**, 마이크 6채널
캡처 시작, STT·DOA·TTS가 GPU에 상주. 입력창 타이핑→Enter→**스피커 발화(TTS 라이브 확인)**, ESC 정상 종료.
STT+TTS 동시 상주는 스모크 테스트로 OOM 없음 확인(§4).

⚠️ **아직 로그로 남기지 않은 것:** 마이크 발화 → 자막+화살표가 실시간으로 함께 반응하는 *완전 동시 라이브*.
개별 기능·동시 상주·타이핑→발화는 확인됐고, **완전 동시 라이브는 시연 촬영에서 최종 확인**한다.

## DOA 오차 — 케이스 조립 후 측정 (완료, 2026-07-25)

**결과: 평균 절대 오차 1.5° (네 방향 모두 안정).** `front_offset: 90` 확정.

| 방향 | raw | 화면각 | 퍼짐 | 채택% | 오차 |
|---|---|---|---|---|---|
| 앞 | 89 | 1° | 3° | 100% | 1° |
| 오른쪽 | 358 | 92° | 3° | 100% | 2° |
| 뒤 | 273 | 177° | 2° | 100% | 3° |
| 왼쪽 | 180 | 270° | 0° | 100% | 0° |
| **평균** | | | | | **1.5°** |

- raw가 89·358·273·180 → 정확히 90° 간격. 퍼짐 0~3°로 매우 안정, 채택 100%.
- 측정 로그: `doyu/tests/2026-07-25_doa-phase8.txt` (3번째 블록, 00:53:57).

### ★ 핵심 발견 — 케이스는 마이크 면을 반드시 노출해야 한다
케이스 조립 직후, 마이크 면(LED·마이크 구멍 4개)이 케이스에 덮인 상태에서는 DOA가 **완전히 망가졌다**:
- 평균 오차 28~64°, 게다가 **잴 때마다 다르게** 나옴(앞이 raw 90→270으로 180° 뒤집힘).
- 원인: 마이크 4개에 소리가 **직접 도달하지 못하고 케이스 안 반사·회절음**을 잡아 방향이 엉킴.
- **마이크 면을 노출하자 즉시 평균 1.5°로 회복**(케이스 전 6°보다도 좋음).

→ **설계 요구사항:** 최종 케이스는 마이크 배열 윗면(구멍 4개)을 **반드시 밖으로 노출**하거나 그 위에
**개구부/그릴**을 두어야 한다. 마이크가 케이스에 파묻히면 방향 기능이 무력화된다.

### 측정 도구 (개선판, `src/audio/doa.py` `_selftest`, Phase 8)
- 4초간 raw 각도를 촘촘히 모아 **원형 최빈 클러스터**(`_robust_center`)로 반사음·잡값 제거.
- 방향마다 **안정/불안정 판정**(퍼짐 ≤25°, 채택 ≥60%), 불안정하면 즉시 재측정.
- 평균 오차는 **안정된 방향만** 집계. `front_offset`은 건드리지 않음(측정 전용).
- 절차: `.venv/bin/python src/audio/doa.py` → 앞/오/뒤/왼 각 방향에 정확히 서서 4초간 크게 발화.
